# IPTV频道列表更新需求文档

## 1. 项目背景

需要基于新的北京联通IPTV频道列表（代号：**新**）更新现有的 `iptv.bj.m3u8` 文件（代号：**旧**）。

- **旧文件路径**: `/Users/goodspeedwang/Document/code/private/goodspeedwang.github.io/iptv.bj.m3u8`
- **新文件来源**: https://gist.githubusercontent.com/sdhzdmzzl/93cf74947770066743fff7c7f4fc5820/raw/2ce1dfe71ae7a5b36677f94d5f4ead165d3d46ea/bj-unicom-iptv.m3u

## 2. 文件格式对比

### 2.1 旧文件格式（完整格式）
```m3u8
#EXTM3U x-tvg-url="http://epg.51zmt.top:8000/e.xml.gz"
#EXTINF:-1 tvg-id="1" tvg-name="CCTV1" tvg-logo="http://epg.51zmt.top:8000/tb1/CCTV/CCTV1.png" group-title="[高清]央视",CCTV-1综合[1][高清]
http://192.168.100.1:5140/rtp/239.3.1.129:8008
```

**特点**:
- 包含完整的EXTINF元数据：tvg-id、tvg-name、tvg-logo、group-title
- 频道显示名称规范（如 `CCTV-1综合[1][高清]`）
- URL已添加uporxy代理层：`http://192.168.100.1:5140/rtp/{ip}:{port}`

### 2.2 新文件格式（简化格式）
```
#EXTINF:-1,北京卫视4K
rtp://239.3.1.22:8001
#EXTINF:-1,云南卫视
rtp://239.3.1.26:8108
```

**特点**:
- 仅包含简单的频道名称
- 使用原始rtp地址，缺少uporxy代理层
- 缺少完整的EXTINF元数据（tvg-id、tvg-logo、group-title等）

## 3. 功能需求

### 3.1 核心需求
1. **频道匹配**: 根据rtp地址（239.3.1.x:xxxx）匹配新旧频道
2. **EXTINF信息补全**: 为新频道补充完整的EXTINF元数据
3. **URL转换**: 将rtp地址转换为uporxy格式
4. **group-title整理**: 重新设计分组规则，采用"品质\|类型"两级结构
5. **新增频道标识**: 新列表中独有的频道添加 `x-add="true"` 属性

### 3.2 详细需求

#### 3.2.1 频道匹配规则
- 以rtp地址（`239.3.1.xxx:xxxx`）作为唯一标识进行匹配
- 新列表中存在但旧列表中不存在的频道，需要新增
- 旧列表中存在但新列表中不存在的频道，需要标记或删除

#### 3.2.2 EXTINF信息来源
| 字段 | 来源 | 说明 |
|------|------|------|
| tvg-id | 旧文件 | 保持原有ID |
| tvg-name | 旧文件/网络查询 | 优先从旧文件获取，缺失的需查询EPG数据库 |
| tvg-logo | 旧文件/网络查询 | 格式：`http://epg.51zmt.top:8000/tb1/{分类}/{频道名}.png` |
| group-title | 旧文件 | 保持原有分组（如 `[高清]央视`、`[高清]卫视` 等） |
| 显示名称 | 旧文件 | 格式如 `CCTV-1综合[1][高清]` |

#### 3.2.3 URL转换规则
```
原始地址: rtp://239.3.1.130:8004
转换后:   http://192.168.100.1:5140/rtp/239.3.1.130:8004
```
转换公式：`rtp://{ip}:{port}` → `http://192.168.100.1:5140/rtp/{ip}:{port}`

## 4. 数据处理策略

### 4.1 频道分类规则（新版group-title设计）

**设计原则**：简洁清晰，去掉冗余的中括号，采用"品质|类型"两级结构

| group-title | 说明 | 包含频道 |
|-------------|------|----------|
| `高清\|央视` | 央视高清频道 | CCTV1-17、CCTV5+、CGTN系列 |
| `高清\|卫视` | 各地方卫视高清 | 湖南卫视、江苏卫视、浙江卫视等 |
| `高清\|北京` | 北京本地高清 | BRTV系列、北京卫视等 |
| `高清\|数字` | 数字/专业频道高清 | 金鹰纪实、快乐垂钓、茶频道等 |
| `高清\|IPTV` | IPTV专属高清频道 | IPTV热播剧场、睛彩系列、淘系列等 |
| `高清\|购物` | 购物频道高清 | 聚鲨环球、优购物、家有购物等 |
| `高清\|区县` | 北京各区县频道 | 朝阳融媒、密云电视台、房山电视台等 |
| `4K` | 4K超高清频道 | 北京卫视4K、CCTV 4K、爱上4K等 |
| `标清\|央视` | 央视标清（如有） | - |
| `标清\|卫视` | 各地方卫视标清 | 广西卫视、陕西卫视等 |
| `标清\|北京` | 北京本地标清 | BRTV系列标清 |
| `标清\|数字` | 数字频道标清 | 山东教育、财富天下等 |
| `标清\|IPTV` | IPTV专属标清 | - |
| `少儿` | 儿童/少儿频道（跨清晰度） | 卡酷少儿、金鹰卡通、嘉佳卡通等 |

**分类判断逻辑**：
1. **央视**: tvg-name以"CCTV"或"CGTN"开头
2. **卫视**: tvg-name包含"卫视"且非北京卫视
3. **北京**: tvg-name包含"北京"、"BRTV"、"BTV"
4. **数字**: 专业付费频道（快乐垂钓、茶频道等）
5. **IPTV**: 名称以"IPTV"开头或睛彩/淘系列
6. **购物**: 名称包含"购物"、"聚鲨"
7. **区县**: 名称包含"朝阳"、"密云"、"房山"、"通州"、"延庆"、"门头沟"
8. **4K**: 名称包含"4K"
9. **少儿**: 名称包含"少儿"、"卡通"、"动画"、"baby"

## 8. 详细处理规则

### 8.1 频道名称映射规则

**处理原则**：根据语义判断，可合并或区分

| 场景 | 处理方式 | 示例 |
|------|----------|------|
| 同一频道，名称简化 | 采用更完整的名称 | `北京卫视` → `BRTV北京卫视` |
| 同一频道，品牌升级 | 采用新品牌名称 | `BTV文艺` → `BRTV文艺` |
| 不同清晰度版本 | 保留区分标识 | `北京卫视` vs `北京卫视4K` |
| 地区性差异 | 保留地区标识 | `北京卫视SDR` vs `北京卫视HDR` |

**频道名称标准化**：
- 统一使用 `BRTV` 前缀表示北京广播电视台频道
- 清晰度标识放在名称后：`[高清]`、`[标清]`、`[4K]`
- 频道号保留（如有）：`[1]`、`[121]` 等

### 8.2 重复频道去重规则

**判断依据**：rtp地址（239.3.1.xxx:xxxx）完全相同视为重复

**去重策略**：
1. 优先保留有完整EXTINF信息的记录（旧文件）
2. 多个新记录冲突时，保留第一个出现的
3. 去重后记录日志，列出被合并的重复项

**重复示例**：
```
新列表中发现重复:
- 优购物高清: 239.3.1.190:8001, 239.3.1.191:8001
- 家有购物高清: 239.3.1.185:8001, 239.3.1.186:8001
```

### 8.3 EPG信息查询策略

**查询优先级**：
1. **第一优先**：旧文件中的已有数据（tvg-id, tvg-name, tvg-logo）
2. **第二优先**：EPG数据库（http://epg.51zmt.top:8000）
3. **第三优先**：网络搜索补充（百度百科、官方信息等）
4. **最后**：无法获取的字段留空

**字段留空规则**：
| 字段 | 无法获取时的处理 |
|------|-----------------|
| tvg-id | 留空 `""` |
| tvg-name | 使用新列表中的频道名称（清理问号等异常字符） |
| tvg-logo | 留空 `""` |
| group-title | 按分类规则确定，不依赖EPG |

### 8.4 分组边界模糊处理

**模糊场景及处理**：

| 模糊场景 | 判断依据 | 默认分组 |
|----------|----------|----------|
| IPTV专属 vs 数字频道 | 名称含"IPTV"前缀或"睛彩/淘"系列 → IPTV；其他专业频道 → 数字 | 不确定时用 `高清\|数字` |
| 购物频道识别 | 名称明确含"购物"→购物；名称模糊（如"聚鲨"）→查EPG确认 | 不确定时用 `高清\|数字` |
| 区县台识别 | 名称含"朝阳"、"密云"、"房山"等北京区县名 → 区县 | 不确定时用 `高清\|数字` |
| CCTV3高清 vs CCTV3高清回放 | 地址不同(172)视为不同频道，但分组相同 | 都归 `高清\|央视` |

**兜底分组**：`高清|其他` / `标清|其他`
- 当频道无法明确归类到上述任何分组时使用

### 8.5 新增标识准确性规则

**定义**：只有rtp地址在旧列表中完全不存在的频道，才标记为 `x-add="true"`

**判断逻辑**：
```
if 新频道.rtp_address not in 旧文件所有地址:
    标记 x-add="true"
else:
    不标记（视为地址更新或重复）
```

**常见误区**：
- ❌ 旧频道换了新地址 → 这不是新增，是地址变更（旧地址应删除）
- ❌ 同一频道多个清晰度版本 → 每个清晰度视为独立频道
- ✅ 全新频道（如门头沟高清）→ 标记为新增

**示例**：
```m3u8
# CCTV3 - 原有频道，地址从239.3.1.172换到239.3.1.xxx
# 不标记x-add
#EXTINF:-1 tvg-id="3" tvg-name="CCTV3" group-title="高清|央视",CCTV-3综艺

# 门头沟高清 - 全新频道
#EXTINF:-1 tvg-id="" tvg-name="门头沟高清" group-title="高清|区县" x-add="true",门头沟电视台
```

## 9. 实现建议

### 9.1 数据处理流程
1. 解析旧文件，建立 `rtp地址 → 完整频道信息` 的映射表
2. 解析新文件，获取所有rtp地址和原始名称
3. 新文件去重（按rtp地址）
4. 遍历新文件频道：
   - rtp地址存在于旧表 → 复用旧信息，更新group-title
   - rtp地址不存在 → 标记为新增，查询EPG补全信息
5. 按新分组规则归类
6. 输出格式化M3U8

### 9.2 日志输出建议
- 新增频道列表（带x-add标识的）
- 地址变更频道列表（同一频道不同地址）
- 重复频道去重记录
- EPG查询失败记录（留空的字段）
- 使用兜底分组的频道列表
